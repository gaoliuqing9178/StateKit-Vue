# StateKit Visual System

## 设计目标

StateKit 的视觉系统不是为了追求“组件库感”，而是为了让状态界面在真实 SaaS 产品里显得自然、完整、可信。它要同时做到两件事：

- 一眼看出这是某种状态，而不是普通内容卡片。
- 看起来像产品的一部分，而不是设计稿之外临时补的占位页。

当前实现以统一 Shell 为核心，通过 `tone`、`density`、`layout` 和 `category` 来变化视觉效果。

## 核心视觉原则

- 统一的结构语言：所有 Block 都使用同一套媒体区、标题区、描述区和 CTA 区。
- 类别驱动的语义图形：empty、onboarding、loading、error、permission、upgrade、success 各自对应一组抽象媒体形态。
- 轻插画、重信息：图形负责传达语气，但不能盖过标题和动作。
- 背景有层次但不过度装饰：当前样式使用浅色渐变、柔和边框和阴影，而不是高饱和大面积插画。

## Tone 体系

StateKit 当前定义了 5 种 tone，它们不只是颜色切换，而是场景态度。

| Tone | 用途 | 视觉含义 |
| --- | --- | --- |
| `neutral` | 默认说明型状态 | 冷静、中性、不制造额外情绪 |
| `brand` | 引导开始、升级、激活 | 带有产品推动感，强调正向行动 |
| `danger` | 错误、失败、不可用 | 明确告知异常，需要恢复动作 |
| `warning` | 权限、限制、临界风险 | 提醒用户被限制，但不过度惊吓 |
| `success` | 完成、确认、流程闭环 | 给出“已经成功”的结束信号 |

当前 CSS 中这些 tone 主要映射为 accent 色与 soft 背景色：

- `brand`: 蓝色系
- `danger`: 红色系
- `warning`: 橙色系
- `success`: 绿色系
- `neutral`: 深灰蓝系

## Density 体系

`density` 控制的是界面呼吸感，而不是信息层级。

| Density | 适用场景 | 当前表现 |
| --- | --- | --- |
| `compact` | 局部、次级、嵌入式状态 | 更紧凑的内边距和媒体尺寸 |
| `cozy` | 默认状态卡片 | 平衡的信息密度 |
| `spacious` | 首次进入、强引导、整页结果 | 更大的留白与媒体展示 |

使用规则：

- `inline` 常与 `compact` 或 `cozy` 搭配。
- `page` 更适合 `cozy` 或 `spacious`。
- 不要用 `spacious` 去包装一个很小的局部错误，否则会造成视觉失衡。

## Layout 体系

| Layout | 结构特征 | 推荐场景 |
| --- | --- | --- |
| `inline` | 小媒体区 + 内容并排或窄屏堆叠 | 表格、侧栏、局部模块 |
| `panel` | 中尺寸媒体区 + 双栏卡片 | 大多数后台内容区状态 |
| `page` | 居中、大留白、强焦点 | 首次使用、整页失败、强结果页 |

响应式行为：

- 在较窄视口下，`panel` 和 `inline` 会退化为单列堆叠布局。
- CTA 区会在移动端转成纵向堆叠，保证按钮可点性。

## 类别图形语言

当前 `StateBlockShell` 并不是为每个 Block 单独画插图，而是基于类别生成抽象图形：

- `empty`：叠层面板，表达“有结构但没有内容”。
- `onboarding`：路径与目标点，表达“首次进入产品，准备启动第一步”。
- `loading`：旋转环形，表达“系统正在处理”。
- `error`：徽章、交叉线和阴影条，表达“加载或执行失败”。
- `permission`：卡片叠层和锁形，表达“被限制访问”。
- `upgrade`：卡片叠层与 spark 元素，表达“解锁、更高一级能力”。
- `success`：徽章与对勾，表达“任务完成”。

这套做法的好处是统一、稳定、易维护；代价是每个 Block 的个性化程度有限。这是 V1 的有意取舍。

## 文案与视觉配合

视觉系统必须服务于文案，而不是替代文案。当前建议如下：

- `title` 负责把状态说清楚，应该是用户扫一眼就能理解的短句。
- `description` 负责解释原因、后果或下一步。
- `kicker` 当前自动显示类别名，例如 `Error`、`Upgrade`，它是辅助信息，不应承担主要语义。
- CTA 文案应与视觉语气一致，避免出现颜色是 `danger`、按钮却像营销按钮的违和感。

## CTA 规范

当前组件最多显示两个动作按钮。

推荐规则：

- 主按钮提供最推荐的恢复或推进动作。
- 次按钮提供低成本替代路径，例如返回、查看说明、联系支持。
- 如果 preset 默认给了两个按钮，但当前场景只需要一个 CTA，应该显式传 `null` 移除多余按钮，而不是保留无意义动作。
- `loading` 动作应优先通过 `loadingLabel` 提供更具体的进行中文案，而不是一律依赖默认的 `Working...`。
- `disabled` 动作需要保持明显灰态，并通过 `not-allowed` 或 `progress` cursor 明确告知当前不可点击。
- 如果场景没有明确行动，宁可只保留一个动作，也不要机械凑两个按钮。

避免：

- 主次按钮都写成长句。
- 两个按钮语义重叠，例如 `Retry` 与 `Reload now`。
- 在 error 或 permission 状态中放过度强硬的商业 CTA。

## 视觉 Do / Don't

Do：

- 保持浅背景、清晰边界、柔和阴影的产品化基调。
- 用 tone 强化情绪，而不是完全依赖标题。
- 让媒体图形、文案和按钮形成一个完整动作建议。

Don't：

- 不要把 StateKit 当成营销 Banner 系统。
- 不要在 V1 中引入复杂插画或高定制 hero 图，破坏统一性。
- 不要为了“更有设计感”而牺牲状态识别速度。

## V1 之后可扩展的方向

- 为特定 Block 提供可选的专属媒体插画。
- 暴露更细粒度的 design token。
- 允许品牌方更明确地覆盖 tone 的色板与背景风格。
- 在不失控的前提下引入更丰富的 slot 能力。
